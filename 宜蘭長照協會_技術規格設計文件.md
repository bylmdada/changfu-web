# 社團法人宜蘭縣長期照護及社會福祉推廣協會
## 官方網站技術規格與系統設計文件

---

## 前言

本文件為協會官方網站的系統設計規格書，目標是建置一個超越傳統 CMS「文章官網」的結構化資料平台，具備明確的資料模型、友善的後台管理介面，以及可擴充的 API 架構。

---

## A. 前後端技術與架構建議

### A.1 前端架構要件

| 要件類別 | 說明 |
|---------|------|
| 響應式設計（RWD） | 所有頁面須支援桌機、平板、手機等不同螢幕尺寸 |
| 元件化設計 | 將 UI 拆分為可重用元件：Header、Footer、導覽選單、卡片、列表、按鈕、表單等 |
| 頁面類型區分 | 明確區分「靜態頁」與「動態列表頁／詳情頁」 |
| SEO 友善 | 支援伺服器端渲染（SSR）或預先產生靜態頁面（SSG），確保搜尋引擎可索引 |
| 無障礙設計 | 符合 WCAG 2.1 AA 等級，確保長者與身心障礙者可使用 |
| 效能優化 | 圖片延遲載入、CSS/JS 壓縮、快取策略 |

### A.2 前端頁面類型

| 頁面類型 | 說明 | 範例 |
|---------|------|------|
| 靜態單頁 | 內容固定或極少更新，直接渲染 | 首頁、協會簡介、宗旨願景 |
| 資料列表頁 | 從資料庫抓取多筆資料，支援篩選排序 | 服務據點列表、最新消息列表、課程列表 |
| 資料詳情頁 | 顯示單筆資料完整內容 | 據點詳情、新聞詳情、課程詳情 |
| 表單頁 | 使用者輸入資料並提交 | 聯絡我們、線上報名、線上應徵 |

### A.3 必備前端元件清單

```
共用元件
├── Header（網站標頭，含 Logo、主選單、搜尋）
├── Footer（網站頁尾，含聯絡資訊、快速連結、版權宣告）
├── Navigation（主選單，含下拉子選單）
├── Breadcrumb（麵包屑導覽）
├── Card（卡片，用於顯示據點、新聞、課程摘要）
├── List（列表容器，支援分頁與篩選）
├── FilterBar（篩選列，依類別、鄉鎮、日期篩選）
├── Pagination（分頁元件）
├── Button（按鈕，含多種樣式）
├── Form（表單元件，含驗證）
├── Modal（彈出視窗）
├── Map（地圖嵌入，Google Maps）
└── ImageGallery（圖片展示）
```

### A.4 後端與 CMS 要件

| 要件類別 | 說明 |
|---------|------|
| 多內容型態支援 | 可定義多種 Content Types（服務項目、據點、新聞、課程、職缺），各有獨立欄位 |
| 欄位型態豐富 | 支援文字、富文字、數字、日期、選單、多選、圖片、檔案、地址、經緯度等欄位型態 |
| 關聯欄位 | 支援資料間的關聯（如：據點關聯到服務項目） |
| 狀態管理 | 草稿、已發布、已下架等狀態控制 |
| 排程發布 | 可設定發布時間與下架時間 |
| 版本控制 | 保留編輯紀錄，可回復舊版本 |
| 使用者權限 | 區分管理員、編輯者、檢視者等角色 |
| 媒體管理 | 圖片與檔案集中管理，支援資料夾分類 |
| 搜尋功能 | 後台可搜尋所有內容 |
| 多語預留 | 預留多語系擴充架構（繁中為主，未來可擴充英文） |

### A.5 API 架構要件

| 要件 | 說明 |
|------|------|
| RESTful API | 提供標準 REST API，支援 GET/POST/PUT/DELETE |
| 或 GraphQL | 視需求可採用 GraphQL 提供彈性查詢 |
| 認證機制 | API 需有 Token-based 認證（JWT 或 API Key） |
| 分頁與篩選 | API 支援分頁參數（page, limit）與篩選參數（filter） |
| 文件化 | 提供 OpenAPI/Swagger 文件 |
| CORS 設定 | 允許前端跨域請求 |

## C. 網站資訊架構（IA）與功能設計

### C.1 主選單階層結構

```
首頁 [靜態頁]
│
├── 關於我們
│   ├── 協會簡介 [靜態頁]
│   ├── 宗旨與願景 [靜態頁]
│   ├── 組織架構 [靜態頁]
│   └── 理監事名單 [靜態頁]
│
├── 核心業務
│   ├── 長者照顧服務 [靜態介紹頁 + 據點列表]
│   │   └── {據點詳情頁} [動態詳情頁，由 ServiceLocation 資料產生]
│   ├── 失智社區服務據點 [靜態介紹頁 + 據點列表]
│   │   └── {據點詳情頁} [動態詳情頁]
│   └── 輔具資源服務 [靜態介紹頁 + 據點詳情]
│
├── 最新消息 [資料列表頁，由 News 資料產生]
│   ├── 篩選：全部 / 公告 / 新聞 / 活動花絮
│   └── {新聞詳情頁} [動態詳情頁]
│
├── 課程與訓練 [資料列表頁，由 Course 資料產生]
│   ├── 篩選：全部 / 照顧訓練 / 照顧者支持 / 專業研習
│   └── {課程詳情頁} [動態詳情頁]
│
├── 招募與徵才 [資料列表頁，由 Job 資料產生]
│   ├── 篩選：全部 / 全職 / 兼職
│   └── {職缺詳情頁} [動態詳情頁]
│
├── 資源與下載 [資料列表頁，由 Resource 資料產生]
│   └── 篩選：表單 / 政策資訊 / 法規 / 衛教資料
│
└── 聯絡我們
    ├── 協會與據點資訊 [靜態頁 + 據點列表]
    ├── 交通與地圖 [靜態頁 + 嵌入地圖]
    └── 線上聯絡表單 [表單頁]
```

## D. 後台操作與維運動線建議

### D.1 後台介面架構

```

├── 快速統計：本月新增消息、課程、訪客人數

│
```

### D.2 典型操作流程

#### 流程 1：新增一個樂智據點

1. 登入後台，從側邊選單點選「服務據點管理」
2. 點擊「新增據點」按鈕
3. 填寫表單欄位：
   - 據點名稱：輸入「羅東樂智據點」
   - 據點類型：下拉選擇「樂智據點」
   - 關聯服務項目：選擇「失智社區服務據點」
   - 所在鄉鎮：下拉選擇「羅東」
   - 完整地址：輸入地址
   - 服務時間：輸入「每週二、四 09:00-12:00」
   - 聯絡電話：輸入電話
   - 服務內容：使用富文字編輯器輸入詳細說明
   - 上傳環境照片
4. 點擊「儲存為草稿」或「立即發布」
5. 預覽前台頁面確認無誤

#### 流程 2：開設一門課程

1. 從側邊選單點選「課程與訓練管理」
2. 點擊「新增課程」
3. 填寫表單：
   - 課程名稱
   - 類別：下拉選擇「照顧訓練」
   - 上課日期與時間
   - 上課地點：可選擇關聯據點或輸入其他地點
   - 講師
   - 名額限制
   - 報名截止日
   - 報名方式：輸入說明或貼上 Google 表單連結
   - 課程說明內容
   - 上傳課程簡章（附件）
4. 設定發布狀態
5. 儲存並預覽

#### 流程 3：發布一則徵才公告

1. 從側邊選單點選「招募職缺管理」
2. 點擊「新增職缺」
3. 填寫表單：
   - 職缺名稱
   - 工作地點：選擇關聯據點或輸入
   - 類型：下拉選擇「全職」
   - 工作內容：使用富文字編輯器
   - 資格條件
   - 薪資範圍（選填）
   - 應徵方式
   - 截止日期
   - 勾選「急徵」（若需要）
4. 發布

#### 流程 4：處理聯絡表單

1. 從側邊選單點選「聯絡表單記錄」
2. 查看未讀表單列表（以粗體或標籤標示）
3. 點擊進入查看詳情
4. 記錄處理備註
5. 標記為「已讀」或「已回覆」

### D.3 介面設計建議

| 元素 | 建議 |
|-----|------|
| 側邊選單 | 固定於左側，以圖示+文字呈現，可摺疊 |
| 列表頁 | 表格形式，顯示標題、狀態、日期、操作按鈕 |
| 編輯頁 | 單欄或雙欄表單，右側可預覽 |
| 富文字編輯器 | 提供基本格式、圖片插入、連結功能 |
| 圖片上傳 | 支援拖放上傳，自動壓縮與縮圖產生 |
| 狀態標籤 | 以顏色區分草稿（灰）、已發布（綠）、已下架（紅） |
| 批次操作 | 支援勾選多筆批次發布或刪除 |
| 確認提示 | 刪除或重要操作需二次確認 |

### D.4 使用者角色建議

| 角色 | 權限說明 |
|-----|---------|
| 系統管理員 | 全部權限，包含使用者管理與系統設定 |
| 內容編輯者 | 管理所有內容（新聞、課程、據點等），無系統設定權限 |
| 據點管理員 | 僅能管理特定據點相關內容 |
| 檢視者 | 僅能檢視後台資料，無編輯權限 |

## 附錄：技術實作建議方案（參考）

以下為可能的技術選型，供團隊評估：

### 方案 A：Headless CMS + 前端框架

- **CMS**：Strapi / Directus / Sanity / Contentful
- **前端**：Next.js / Nuxt.js
- **優點**：資料結構彈性高、API 原生支援、前端效能佳
- **適合**：有前端開發資源的團隊

### 方案 B：WordPress + 自訂欄位外掛

- **CMS**：WordPress + ACF（Advanced Custom Fields）
- **前端**：WordPress 佈景主題或 Headless 模式
- **優點**：生態系成熟、非技術人員易上手
- **適合**：偏好傳統 CMS 介面的團隊

### 方案 C：純靜態網站 + JSON 資料

- **前端**：純 HTML/CSS/JavaScript
- **資料**：JSON 檔案作為資料來源
- **優點**：無需後端伺服器、成本最低
- **適合**：小型網站、更新頻率低
